## 2. 产品概述

### 2.1 产品定位

Mycel Mesh 是一款基于 WireGuard 协议的轻量级虚拟组网工具，旨在将分布在不同地理位置的设备和服务器无缝连接到一个虚拟局域网中，实现设备间的点对点安全通信。

### 2.2 产品愿景

> 让虚拟组网像使用局域网一样简单，随时随地，无缝连接。

### 2.3 目标用户

| 用户群体 | 使用场景 | 核心诉求 | 典型用例 |
|----------|----------|----------|----------|
| 开发者 | 远程开发环境访问、多设备调试 | 低延迟、易配置 | 本地 VS Code 远程连接云服务器开发环境 |
| 运维工程师 | 服务器集群管理、远程维护 | 安全性、稳定性 | 不暴露 SSH 端口到公网，安全运维 |
| 游戏玩家 | 局域网联机游戏 | 低延迟、易分享 | 我的世界/联机游戏虚拟局域网 |
| 小微企业 | 分布式办公网络互联 | 成本低、易部署 | 总部 + 分店 ERP/文件共享系统 |
| IoT 开发者 | 设备远程管理和数据采集 | 批量管理、低功耗 | 远程 IoT 设备监控和固件升级 |

## 3. 需求背景

### 3.1 用户场景故事 (User Stories)

**场景 1：远程开发**
> 小王是后端开发，公司在阿里云有测试服务器。之前需要开放 22 端口到公网，经常被扫描攻击。使用 Mycel Mesh 后，服务器只需出站连接，本地和设备都能安全访问内网服务。

**场景 2：多设备调试**
> 小李做 IoT 开发，办公室有开发板、测试机，回家后需要调试设备。以前要配置端口映射，现在一键加入虚拟网络，就像在办公室局域网一样。

**场景 3：游戏联机**
> 小张想和朋友玩联机游戏，但不在同一网络。使用 Mycel Mesh 创建虚拟房间，朋友加入后直接通过虚拟 IP 联机，无需公网 IP。

### 3.2 痛点分析

| 痛点 | 描述 | 现有方案问题 |
|------|------|--------------|
| 地域限制 | 设备和服务器分布在不同网络环境 | 传统 VPN 配置复杂 |
| 内网穿透困难 | NAT 环境下设备无法直接通信 | 需要公网 IP 或复杂端口映射 |
| 安全风险 | 暴露在公网的服务易受攻击 | 需要额外的安全防护 |
| 运维成本高 | 多设备连接管理繁琐 | 缺乏统一的管理界面 |

### 3.2 竞品分析

| 产品 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|--------------|
| Tailscale | 开箱即用、稳定 | 闭源、依赖官方服务器、免费版限制多 | 自建可控、无设备数限制 |
| ZeroTier | 跨平台、P2P | 配置复杂、Moon 节点需手动 | 自动化程度更高 |
| Netmaker | 开源、功能全 | 部署复杂、学习成本高 | 轻量化、单一二进制 |
| FRP/内网穿透 | 简单 | 点对点需中转、带宽瓶颈 | 真正 Mesh 直连 |

### 3.3 市场机会

- WireGuard 协议相比 OpenVPN 性能提升 3-4 倍
- 代码量更少（4000 行 vs 40 万行），安全性更高，审计更容易
- 现代内核支持（Linux 5.6+ 内置），性能优异
- 自组网需求增长：远程办公、分布式团队、IoT 设备管理

### 4.3 API 设计

#### 认证 API

```yaml
POST /api/v1/auth/login:
  描述：管理员登录
  请求:
    username: string
    password: string
  响应:
    token: string (JWT)
    expires_in: number (3600)

POST /api/v1/auth/refresh:
  描述：刷新访问令牌
  请求:
    refresh_token: string
  响应:
    token: string
    expires_in: number
```

#### 节点管理 API

```yaml
GET /api/v1/nodes:
  描述：获取节点列表
  响应:
    nodes:
      - id: string
        name: string
        private_ip: string
        public_key: string
        status: enum (online/offline)
        last_seen: timestamp
        latency_ms: number

POST /api/v1/nodes:
  描述：注册新节点
  请求:
    name: string
    public_key: string
    auth_token: string (加入令牌)
  响应:
    node_id: string
    assigned_ip: string
    peers: [PeerConfig]

GET /api/v1/nodes/{id}:
  描述：获取节点详情
  响应:
    node: Node
    traffic_stats: TrafficStats

DELETE /api/v1/nodes/{id}:
  描述：删除节点
  响应:
    success: boolean
```

#### 网络配置 API

```yaml
GET /api/v1/network:
  描述：获取网络配置
  响应:
    network:
      id: string
      name: string
      subnet: string
      node_count: number

PUT /api/v1/network:
  描述：更新网络配置
  请求:
    name: string
    subnet: string

GET /api/v1/network/acl:
  描述：获取访问控制列表
  响应:
    rules:
      - id: string
        source: string (节点 ID 或 CIDR)
        destination: string
        action: enum (allow/deny)
        ports: number[]
```

## 5. 技术架构

### 5.1 技术选型

| 组件 | 技术选型 | 选型理由 | 备选方案 |
|------|----------|----------|----------|
| 核心协议 | WireGuard | 高性能、代码少、安全性高 | OpenVPN（太重）、IPsec（太复杂） |
| 后端语言 | Go | 跨平台编译、并发模型好、WireGuard 有官方 Go 实现 | Rust（学习曲线陡）、C++（开发效率低） |
| 前端框架 | React + TypeScript | 生态成熟、类型安全 | Vue（生态稍弱）、Svelte（生态新） |
| 数据库 | SQLite (单节点) / PostgreSQL (高可用) | 简单场景零依赖、复杂场景可扩展 | MySQL（过重）、etcd（过于复杂） |
| 信令服务 | gRPC | 高性能、双向流、强类型 | WebSocket（弱类型）、HTTP SSE（单向） |
| NAT 穿透 | STUN + Port Mapping | 标准协议、成功率高 | 仅中继（带宽成本高） |
| 打包部署 | Docker + systemd | 覆盖云和本地场景 | K8s（过重） |

### 12.5 术语表

| 术语 | 解释 |
|------|------|
| WireGuard | 现代高性能 VPN 协议，使用 ChaCha20-Poly1305 加密 |
| NAT 穿透 | 穿越网络地址转换建立直连，常用 STUN/TURN 协议 |
| 协调服务器 | 负责节点发现、密钥交换、配置分发的中枢服务 |
| Mesh Network | 网状网络，节点间可直连，无需全部流量经过中枢 |
| Exit Node | 出口节点，其他节点可通过其访问公网 |
| STUN | Session Traversal Utilities for NAT，NAT 穿透协议 |
| TURN | Traversal Using Relays around NAT，中继转发协议 |
| CIDR | 无类别域间路由，如 10.0.0.0/24 表示 256 个 IP |
| ACL | 访问控制列表 (Access Control List)，控制节点间访问权限 |
